# SQL-анализ отзывов о банках

Анализ 8 472 отзывов о 71 банке (данные banki.ru, собраны собственным парсером)
на чистом SQL: от нормализации сырой таблицы до оконных функций.

## Данные

- **8 472 отзыва**, **71 банк**.
- Разметка тональности: `y = 0` — негатив (737), `y = 1` — позитив (7 735).
- Поля: `rating` (оценка пользователя 1–5), `text`, `title`, `review_date`.

### Ограничения данных

- **`review_date` — время сбора парсером**, а не дата публикации отзыва
  (весь диапазон укладывается в ~4 дня с секундной точностью). Поэтому
  временной анализ (тренды, динамика) не проводился — на этих данных он был бы
  недостоверным.
- **Классы сильно несбалансированы (~1:10)**, что отражает реальное
  преобладание позитивных отзывов в источнике. Учтено при интерпретации:
  доля негатива считается относительно среднего по выборке, а не как
  абсолютная величина.

## Схема

Сырые данные (`reviews_raw`, одна плоская таблица из parquet) нормализованы
в две связанные таблицы:

```
banks(bank_id PK, bank_name)
reviews(review_id PK, bank_id FK -> banks, review_date, rating, y, text, title)
```

Разделение на справочник банков и таблицу отзывов задаёт связь по ключу
`bank_id` вместо дублирования названия банка в каждой строке.

## Структура

- `schema.sql` — создание таблиц и нормализация из сырой таблицы (`INSERT ... SELECT ... JOIN`).
- `load_data.py` — загрузка parquet в PostgreSQL.
- `queries.sql` — 15 запросов: агрегации, JOIN, подзапросы, CTE, оконные функции.

## Что показал анализ

- Негативные отзывы в среднем заметно длиннее позитивных — недовольные
  пользователи подробнее описывают проблему (запрос 3).
- Часть отзывов показывает **расхождение оценки и тональности**: высокий
  `rating` при негативной разметке (и наоборот) — кандидаты на сарказм или
  ошибки авторазметки (запрос 6).
- По ряду банков **средний rating и доля позитива расходятся в рангах**:
  метрики удовлетворённости не всегда согласованы, одна может завышать
  картину относительно другой (запрос 11).
- Банки распределяются по доле негатива на «проблемные / средние / хорошие»
  — видно, какая часть источника формирует основной негативный фон
  (запросы 9–10).

## Стек

PostgreSQL 17, Python (pandas) для загрузки данных.
